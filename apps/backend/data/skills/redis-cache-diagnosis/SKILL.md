# Redis 缓存诊断

## 任务目标

- 用只读 Redis 查询确认异常键模式、配置值、TTL 或键缺失问题，并给出可验证的根因判断。

## 运行上下文

- 该 skill 在 Chronos 服务端容器内执行，不依赖用户本机环境。
- 优先使用 Redis MCP；如需额外工具，只能在说明用途后使用 `runContainerCommand`。
- 键空间探测必须先收敛 pattern，不要直接做无边界全量扫描。

## 推荐流程

1. 确认项目中存在 Redis 服务，并激活 MCP。
2. 根据告警信息提取 key pattern、业务前缀或热点对象，不要先猜固定键名。
3. 先用 pattern 枚举和样本读取确认键类型、TTL、值结构和异常分布。
4. 对配置异常问题，比较正常样本与异常样本；对缓存缺失问题，确认是否是 TTL、淘汰或根本未写入。
5. 仅在证据充分后才输出根因。

## 查询策略

- 优先使用 pattern + 小样本，不要把某一个键值当成全局结论。
- 先确认键类型和 TTL，再解释值内容，避免把已过期、列表或 hash 错当成字符串。
- 只有在告警明确指向限流、功能开关或缓存污染时，才按相应前缀下钻。
- 如果需要读取大 key，先说明风险并控制样本量。

## 风险边界

- 默认只读，不执行 `SET`、`DEL`、`EXPIRE`、`FLUSHDB` 或其他写操作。
- 如果 Redis 连接、认证或 db 选择信息不足，直接报告阻塞。

## 输出要求

- 最终回复必须写明：目标 key pattern、关键键样本或 TTL 证据、根因判断，以及是否已经激活 MCP。
