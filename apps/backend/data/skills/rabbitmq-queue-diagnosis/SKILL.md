# RabbitMQ 队列诊断

## 任务目标

- 用只读 RabbitMQ 查询确认积压队列、消费者状态、流控或死信问题，给出可验证的根因判断。

## 运行上下文

- 该 skill 在 Chronos 服务端容器内执行，不依赖用户本机环境。
- 优先使用 RabbitMQ MCP；如需补充能力，只能在说明用途后使用 `runContainerCommand`。
- 先定位异常队列和节点，再分析消费者、连接和消息堆积。

## 推荐流程

1. 确认项目中存在 RabbitMQ 服务，并激活 MCP。
2. 先查看节点和队列概览，锁定消息量异常、无消费者或高未确认数的队列。
3. 对异常队列查看消费者、连接、通道和入队/出队速率。
4. 如果存在死信队列，再查看死信流向和积压变化。
5. 用队列深度、消费者状态和节点资源共同支撑结论。

## 查询策略

- 先看队列和节点总览，再下钻到单个消费者或连接，不要直接展开全部连接细节。
- 区分“生产过快”“消费过慢”“消费者断连”“节点流控”和“死信异常”几类问题。
- 关注 `messages_ready`、`messages_unacknowledged`、consumers、连接数和资源告警之间的组合关系。
- 如果问题可能在消费应用而不是 RabbitMQ 本身，要在结论中明确区分。

## 风险边界

- 默认只读，不消费、发布、删除消息，也不修改队列、交换机、绑定或策略。
- 如果连接串、凭据或管理权限不足，直接报告阻塞。

## 输出要求

- 最终回复必须写明：异常队列或节点、关键消息或消费者证据、根因判断，以及是否已经激活 MCP。
