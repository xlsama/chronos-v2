# Sentry 错误分析方法论

## 适用场景

- 应用异常率突增
- 新版本发布后出现未捕获异常
- 性能回归检测
- 用户反馈的特定错误排查
- 错误趋势分析

## 诊断步骤

### 1. 查看项目概览

- 列出 Sentry 项目，找到与告警相关的项目
- 检查近期错误统计和趋势
- 关注新出现的（first seen）和突增的错误

### 2. 分析错误详情

- 获取热门 Issues 列表，按出现频率排序
- 查看每个 Issue 的详细信息：
  - 错误类型和消息
  - 堆栈跟踪（定位到具体代码行）
  - 首次和最近出现时间
  - 影响的用户数

### 3. 检查错误上下文

- 查看错误发生时的 breadcrumbs（操作路径）
- 检查请求参数、用户信息、设备信息
- 分析 tags 和 extra data

### 4. 分析趋势

- 对比不同时间段的错误数量
- 与版本发布时间线关联
- 检查是否有特定环境（production/staging）集中

### 5. 输出诊断结论

- 列出关键错误及其影响范围
- 提供堆栈跟踪和根因分析
- 建议修复优先级和方向

## 常见错误类型

| 错误类型 | 可能原因 | 诊断方向 |
|----------|----------|----------|
| TypeError | 空值引用 | 检查输入验证 |
| TimeoutError | 外部服务超时 | 检查依赖服务状态 |
| ConnectionError | 数据库/API 不可达 | 检查网络和连接配置 |
| MemoryError | 内存泄漏 | 检查资源限制和使用模式 |

## 安全注意事项

- 只读操作，不修改 Issue 状态或配置
- 错误详情中可能包含敏感用户数据
