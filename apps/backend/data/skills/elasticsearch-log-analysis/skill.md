# Elasticsearch 日志分析方法论

## 适用场景

- 应用错误日志突增
- 需要按时间范围检索日志
- 分析错误模式和根因
- 性能瓶颈追踪
- 集群健康状态异常

## 诊断步骤

### 1. 检查集群健康

- 查看集群状态（green/yellow/red）
- 检查节点数量和分片分配
- 关注 unassigned_shards 数量

### 2. 定位相关索引

- 列出可用索引，找到与告警相关的日志索引
- 通常按日期命名：`app-logs-2026.03.13`
- 检查索引大小和文档数量

### 3. 搜索错误日志

- 按时间范围搜索 error/critical 级别日志
- 使用关键词过滤（服务名、错误类型、堆栈跟踪）
- 按时间排序，找到最早的错误记录

### 4. 聚合分析

- 按错误类型分组统计出现次数
- 按时间桶（1分钟/5分钟）统计错误趋势
- 按服务名分组，定位错误集中的服务

### 5. 关联分析

- 通过 trace_id 或 request_id 关联相关日志
- 重建请求链路，找到错误源头
- 对比错误发生前后的日志模式变化

### 6. 输出诊断结论

- 总结错误模式和根因
- 提供受影响的时间范围和服务列表
- 建议修复方向

## 常见模式

| 症状 | 根因 | 查询方式 |
|------|------|----------|
| 错误日志激增 | 部署引入 bug | 按时间聚合 + 关联部署事件 |
| 查询超时 | 索引太大 | 检查索引大小和分片 |
| 集群 red | 节点宕机 | 检查集群健康和节点状态 |

## 安全注意事项

- 只执行搜索查询，不执行索引修改或删除操作
- 日志中可能包含敏感信息，注意脱敏
