# MongoDB 运维诊断方法论

## 适用场景

- 查询响应变慢
- 连接数飙升或连接超时
- 副本集同步延迟
- 数据不一致或缺失
- 索引缺失导致全表扫描

## 诊断步骤

### 1. 检查数据库状态

- 使用 `db.serverStatus()` 查看整体状态
- 关注 connections（当前连接数/可用连接数）
- 检查 opcounters（各类操作计数）

### 2. 分析慢查询

- 查看 `system.profile` 集合获取慢查询日志
- 关注 millis > 100ms 的查询
- 检查查询计划是否使用了索引（COLLSCAN 表示全表扫描）

### 3. 检查集合状态

- 使用 `db.collection.stats()` 查看集合大小和索引信息
- 确认索引是否覆盖常用查询模式
- 检查文档数量是否与预期一致

### 4. 检查副本集状态

- 查看副本集成员状态和同步延迟
- 关注 SECONDARY 节点的 optime 延迟
- 检查是否有节点处于 RECOVERING 或 DOWN 状态

### 5. 数据验证

- 针对告警中提到的集合进行数据采样
- 对比预期数据结构和实际文档结构
- 检查必填字段是否有缺失或类型错误

### 6. 输出诊断结论

- 总结根因（索引缺失、连接池配置不当、副本集问题等）
- 列出受影响的集合和业务
- 提供修复建议

## 常见模式

| 症状 | 根因 | 诊断方向 |
|------|------|----------|
| 查询慢 | 缺少索引 | 检查查询计划 |
| 连接超时 | 连接池耗尽 | 检查 connections |
| 数据不一致 | 副本集延迟 | 检查 rs.status() |

## 安全注意事项

- 只执行读操作，不执行 insert/update/delete
- 避免对大集合执行无条件查询
