# SSH 服务器诊断

## 任务目标

- 通过 SSH MCP 在目标主机上执行只读诊断命令，确认系统资源、进程、日志或网络异常的直接证据。

## 运行上下文

- Agent 本身运行在 Chronos 服务端容器内，但诊断命令应通过 SSH MCP 在目标主机上执行，而不是在用户本机执行。
- 只有当 SSH MCP 无法覆盖某个前置检查时，才使用 `runContainerCommand` 检查容器内 SSH 客户端或辅助工具。
- 默认选择非交互、广泛可用的只读命令，不依赖目标主机一定安装某个发行版特有工具。

## 推荐流程

1. 确认项目中存在 SSH 服务，并激活 MCP。
2. 先用轻量命令查看主机概览、负载、内存、磁盘和关键进程，不要一开始就跑重日志检索。
3. 根据症状选择进一步查看进程、监听端口、连接数、系统日志或特定应用日志。
4. 如果命令在目标主机不存在，选择更通用的替代命令，而不是直接失败。
5. 最终用系统指标、进程证据和日志证据共同支撑结论。

## 查询策略

- 先看 `uptime`、`free`、`df`、`ps`、`ss` 这类低成本命令，再决定是否看日志。
- 如果 `journalctl` 不存在，就退回到 `/var/log` 或应用日志文件，不要假设 systemd 一定可用。
- 对网络问题优先看监听端口、连接数和 DNS；对负载问题优先看 CPU/内存 top 进程；对磁盘问题优先看挂载点和大目录。
- 所有命令都应保持只读、短时、非交互。

## 风险边界

- 默认只读，不执行 `rm`、`kill`、`systemctl restart`、`service stop`、包升级或配置修改。
- 如果 SSH 凭据、私钥或网络访问不可用，直接报告阻塞。

## 输出要求

- 最终回复必须写明：目标主机、执行过的关键命令、关键输出证据、根因判断，以及是否已经激活 MCP。
