# Loki 日志分析

## 任务目标

- 用只读 Loki 查询确认异常日志流、关键标签、错误模式和时间线，给出基于日志证据的根因判断。

## 运行上下文

- 该 skill 在 Chronos 服务端容器内执行，不依赖用户本机环境。
- 优先使用 Loki MCP；只有在 MCP 不足时才考虑 `runContainerCommand` 作为辅助。
- 查询时先缩小时间范围和标签范围，避免无界 LogQL 检索。

## 推荐流程

1. 先确认项目中存在 Loki 服务，并激活 MCP。
2. 用告警信息收敛目标服务、namespace、Pod 或容器标签，不要先猜标签名。
3. 先查短时间窗内的错误日志和日志量变化，再扩展到上下文和关联服务。
4. 先做聚合或计数，再看单条日志，避免只凭样本作结论。
5. 如需跨服务串联，优先使用 trace ID、request ID 或已确认标签。

## 查询策略

- 先识别真实标签和值，再写 LogQL；不要假设一定存在 `app`、`namespace` 或 `level` 标签。
- 先做 `count_over_time`、聚合或模式提取，再回看关键时间点日志。
- 将“异常日志增多”和“关键错误首次出现”分开看，前者说明影响面，后者更适合定位根因。
- 日志查询要始终带时间窗和标签，避免全局扫描。

## 风险边界

- 默认只读，不修改日志流配置、采集规则或保留策略。
- 如果缺少 Loki URL、认证信息或相关标签不存在，应明确说明不确定性。

## 输出要求

- 最终回复必须写明：时间范围、目标日志流或标签、关键聚合或日志证据、根因判断，以及是否已经激活 MCP。
